### 3.2. Kiến trúc hệ thống

#### 3.2.1. Tổng quan kiến trúc

Hệ thống VexCode được thiết kế theo mô hình kiến trúc ba tầng (Three-tier Architecture), bao gồm Frontend (React), Backend (Node.js) và Engine (Python). Backend đóng vai trò điều phối trung tâm, nhận yêu cầu từ Frontend, gọi Engine qua subprocess và stream kết quả về client qua SSE.

**Đối chiếu với mô hình TELPA:**

Mô hình TELPA (Technology-Enhanced Language Proficiency Assessment) đã được giới thiệu ở Chương 2 làm khung lý thuyết cho cách tiếp cận hybrid. Trong kiến trúc VexCode, các thành phần của TELPA được ánh xạ như sau:

*Bảng 3.3. Ánh xạ mô hình TELPA vào kiến trúc VexCode*

| Thành phần TELPA | Vai trò | Module tương ứng trong VexCode |
|-----------------|---------|-------------------------------|
| Rule-based Engine | Phân tích cú pháp, phát hiện lỗi theo luật | Scanner (OpenGrep + ISO 25010) |
| Context-aware Engine | Hiểu ngữ cảnh liên file, đồ thị phụ thuộc | Enricher (GitNexus Knowledge Graph) |
| AI-powered Engine | Suy luận ngữ nghĩa, đề xuất sửa chữa | Resolver (LLM) |
| Orchestrator | Điều phối giữa các engine, quản lý luồng | Pipeline + Backend Node.js |
| Output Interface | Trình bày kết quả, tương tác người dùng | React Dashboard |

#### 3.2.2. Các thành phần module

Hệ thống được tổ chức thành ba package độc lập trong cùng một mono-repository:

- **engine/** (Python) — Nhân phân tích mã nguồn: scanner, enricher, resolver, reporter.
- **cli/** (Node.js) — Backend Express + CLI: server, routes, middleware, bridge giao tiếp Python.
- **web/** (React) — Frontend Dashboard: components, hooks, pages, API client.

Các module chức năng chính:

1. **Scanner** — Quét mã nguồn bằng OpenGrep, phát hiện lỗi cú pháp và bảo mật dựa trên pattern-matching. Kết quả được phân loại theo tiêu chuẩn ISO 25010.

2. **Enricher** — Sử dụng GitNexus để trích xuất AST, xây dựng đồ thị tri thức và Call Graph, bổ sung ngữ cảnh cấu trúc cho từng phát hiện.

3. **Resolver** — Gọi API LLM qua đa provider (OpenAI, Anthropic, Google, 9router, NVIDIA), quản lý retry, timeout, token.

4. **Reporter** — Tổng hợp kết quả thành JSON report và SARIF 2.1.0 sidecar.

5. **Dashboard** — Frontend React cung cấp giao diện trực quan để xem kết quả, tương tác với AI và quản lý trạng thái lỗi.

#### 3.2.3. Luồng dữ liệu tổng quan

Người dùng tương tác với Dashboard qua trình duyệt web. Backend nhận yêu cầu, gọi Python Engine để thực thi pipeline phân tích. Engine thực hiện lần lượt bốn giai đoạn, stream kết quả trung gian về Backend, Backend đẩy tiếp về client qua SSE.

Luồng dữ liệu cụ thể:

1. Client gửi yêu cầu quét → Backend nhận request
2. Backend gọi Bridge → Engine thực thi pipeline
3. Engine: Scanner (OpenGrep) → SARIF output
4. Engine: Enricher (GitNexus) → AST + Knowledge Graph
5. Engine: Resolver (LLM) → Phân tích + Đề xuất sửa chữa
6. Engine: Reporter → JSON kết quả hoàn chỉnh
7. Backend nhận kết quả → Stream SSE về client
8. Client cập nhật Dashboard

#### 3.2.4. Thiết kế Pipeline xử lý

Pipeline của VexCode được thiết kế theo mô hình tuần tự bốn giai đoạn. Mỗi giai đoạn nhận đầu ra của giai đoạn trước và bổ sung thêm dữ liệu.

<!-- TODO: Chèn Hình 3.4. Pipeline bốn giai đoạn -->
<!-- Gợi ý: Sơ đồ luồng ngang: Source Code → Scanner → Enricher → Resolver → Reporter → JSON/SARIF -->

```
Source Code
    │
    ▼
┌──────────┐
│ Scanner  │ ← OpenGrep (phân tích tĩnh, pattern-matching trên AST)
│          │ → SARIF output → ISO 25010 classification
└──────────┘
    │ Findings (raw)
    ▼
┌──────────┐
│ Enricher │ ← GitNexus (AST extraction, Knowledge Graph, Call Graph)
│          │ → Code snippet, function context, related files
└──────────┘
    │ Findings (enriched)
    ▼
┌──────────┐
│ Resolver │ ← LLM (OpenAI, Anthropic, Google, 9router, NVIDIA)
│          │ → Phân tích, phân loại, đề xuất sửa chữa
└──────────┘
    │ Findings (resolved)
    ▼
┌──────────┐
│ Reporter │ → JSON report + SARIF 2.1.0 sidecar
└──────────┘
    │
    ▼
  Dashboard + File system
```

**Scanner** — Quét mã nguồn bằng OpenGrep, phát hiện lỗi cú pháp, bảo mật và chất lượng dựa trên pattern-matching trên AST. Kết quả được parse từ SARIF và phân loại theo ISO 25010. Cơ chế fallback sang mock data nếu OpenGrep không khả dụng.

**Enricher** — Bổ sung ngữ cảnh cấu trúc cho từng phát hiện: trích xuất AST bằng tree-sitter, xây dựng Call Graph, lấy nội dung hàm chứa lỗi và các hàm liên quan (callers, callees). Kết quả là enriched findings kèm code snippet và ngữ cảnh liên file.

**Resolver** — Gom nhóm findings theo file, xây dựng prompt cho LLM với ngữ cảnh đã làm giàu, gọi API đến provider đã cấu hình. Hỗ trợ retry tối đa 3 lần, timeout 60 giây. Fallback sang phân tích cơ bản nếu LLM không phản hồi.

**Reporter** — Tổng hợp kết quả thành JSON report với metadata, summary và danh sách findings chi tiết. Đồng thời xuất SARIF 2.1.0 sidecar kèm gợi ý sửa lỗi.
