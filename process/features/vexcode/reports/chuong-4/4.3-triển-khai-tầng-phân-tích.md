## 4.3. Triển khai tầng phân tích

### 4.3.1. Pipeline tổng thể

Pipeline phân tích được điều phối từ file `packages/engine/src/engine/__main__.py`. Module này nhận tham số dòng lệnh từ Node.js bridge, thiết lập logger, và gọi lần lượt bốn phase của pipeline.

Ở phase Scanner, pipeline gọi `scanner.py::run_scan_phase()` với tham số `target`, `use_mock`, và `fast`. Hàm này kiểm tra chế độ fast scan, phát hiện file thay đổi qua git status, và gọi Opengrep binary để quét.

Ở phase Enricher, pipeline gọi `enricher.py::enrich_findings()` với danh sách findings từ scanner. Hàm này kiểm tra GitNexus availability, lấy thông tin repo, và enrich từng finding với AST context sử dụng ThreadPoolExecutor.

Ở phase Resolver, pipeline gọi `resolver.py::resolve_phase()` để tính complexity metrics, chạy naming audit, và gọi AI resolution. Kết quả là danh sách findings đã được bổ sung resolutions.

Ở phase Reporter, pipeline gọi `reporter.py::assemble_report()` và `reporter.py::write_report()` để tổng hợp và ghi file JSON kết quả.

### 4.3.2. Scanner Module

Module `pipeline/scanner.py` xử lý phát hiện fast scan files và điều phối việc quét. Hàm `get_git_state()` kiểm tra xem thư mục đích có phải là git working tree không, lấy commit hash hiện tại và kiểm tra trạng thái dirty. Hàm `_detect_fast_scan_files()` chạy `git status --porcelain` để lấy danh sách file đã thay đổi. Nếu repo sạch (không có thay đổi), trả về mảng rỗng để báo hiệu không cần quét. Nếu có thay đổi, trả về danh sách đường dẫn tuyệt đối của các file đã thay đổi.

Module `core/scanner.py` chứa logic quét với Opengrep. Hàm `run_scan()` gọi binary Opengrep với tham số `--json` để nhận kết quả dạng JSON. Nếu use_mock được bật, hàm trả về mock findings thay vì gọi Opengrep.

### 4.3.3. Enricher Module

Module `pipeline/enricher.py` thực hiện làm giàu findings với AST context từ GitNexus. Hàm `enrich_findings()` kiểm tra GitNexus availability qua `is_gitnexus_available()`. Nếu có, hàm lấy thông tin repo và sử dụng ThreadPoolExecutor với 10 worker để enrich từng finding song song.

Hàm `_enrich_single_finding()` là đơn vị xử lý chính. Với mỗi finding, hàm lấy đường dẫn file và số dòng, gọi `resolve_location_to_symbol()` để xác định symbol tại vị trí lỗi. Nếu tìm thấy symbol, hàm gọi `get_symbol_context()` để lấy thông tin chi tiết gồm symbol name, kind, source code, và danh sách caller. Hàm cũng gọi `get_symbol_impact()` để tính blast radius: danh sách các symbol bị ảnh hưởng nếu sửa lỗi tại vị trí này. Kết quả được gắn vào finding dưới key `ast_context`.

### 4.3.4. Resolver Module

Module `pipeline/resolver.py` thực hiện ba nhiệm vụ trong phase Resolver.

Đầu tiên, `_collect_source_files()` thu thập danh sách file nguồn để phân tích độ phức tạp. Hàm này duyệt thư mục dự án, bỏ qua các thư mục .git, node_modules, .venv, __pycache__, dist, build. Các đuôi file được hỗ trợ gồm .py, .js, .jsx, .ts, .tsx, .java, .go, .rs, .rb. Số file tối đa được giới hạn bởi hằng số MAX_FILES_FOR_COMPLEXITY.

Thứ hai, `_compute_metrics()` gọi Lizard để tính độ phức tạp cyclomatic cho từng file. Kết quả là một dictionary với key là đường dẫn tương đối và value là các chỉ số phức tạp.

Thứ ba, `_run_naming_audit()` chạy AI audit để phát hiện tên biến và hàm khó hiểu. Hàm này lọc các file thuộc dự án (bỏ qua .agents, .claude, .codex, process) và gọi `run_naming_audit()` trong `core/naming_audit.py`.

Sau ba bước trên, hàm `resolve_findings()` được gọi để tạo AI resolution cho từng finding.

### 4.3.5. AI Resolver Module

Module `core/ai_resolver.py` là thành phần quan trọng nhất của pipeline. Module này chịu trách nhiệm gọi API của các mô hình ngôn ngữ lớn để phân tích findings và đề xuất sửa lỗi.

Cấu hình AI được đọc từ file .env thông qua `get_ai_config()`. Config bao gồm provider (openai, anthropic, google, nvidia, 9router), API key, base URL, model name, temperature, và max tokens.

Hàm `resolve_findings()` là điểm vào chính. Hàm này duyệt qua danh sách findings, nhóm theo rule_id để tránh gọi AI trùng lặp cho các finding cùng loại. Với mỗi rule, hàm xây dựng prompt bao gồm: system prompt với hướng dẫn phân tích bảo mật, finding details (file, line, message, severity), surrounding code (10 dòng trước và sau dòng lỗi), và AST context nếu có.

Các prompt template được định nghĩa trong `config/ai_prompts.py` với bốn loại: SYSTEM_PROMPT_RESOLVE cho phân tích và đề xuất sửa lỗi, SYSTEM_PROMPT_ANALYZE cho phân tích chi tiết, SYSTEM_PROMPT_FIX cho sinh code sửa lỗi, và SYSTEM_PROMPT_REVIEW cho review chất lượng bản sửa.

AI resolution chạy song song sử dụng ThreadPoolExecutor với số worker mặc định là 3. Mỗi worker gọi API của provider tương ứng qua HTTP POST với JSON body chứa messages (system + user). Kết quả trả về được parse từ response JSON, bao gồm các trường: suggestion (mô tả cách sửa), remediation_code (mã nguồn thay thế), classification (loại lỗi), và ai_status (success/failed).

Cơ chế retry được triển khai với exponential backoff cho HTTP 429 (rate limit). Nếu tất cả retry đều thất bại, module trả về mock resolution thay vì báo lỗi.

### 4.3.6. ISO 25010 Classification

Module `config/iso25010_taxonomy.py` triển khai bộ phân loại findings theo tiêu chuẩn ISO/IEC 25010. Bộ phân loại sử dụng chiến lược deterministic, không sử dụng ML hay fuzzy matching, để đảm bảo kết quả có thể kiểm tra được.

Thứ tự ưu tiên phân loại như sau: metadata trực tiếp từ rule (nếu có) được ưu tiên cao nhất. Nếu không có, CWE ID được tra cứu trong bảng CWE_TO_ISO25010. Nếu không có CWE, OWASP ID được kiểm tra (tất cả OWASP đều map vào Security). Tiếp theo, rule_id được kiểm tra với context hints (style, naming, convention) để gán vào Maintainability. Sau đó, rule_id được so khớp với danh sách keyword đã sắp xếp theo độ dài giảm dần để tránh false positive. Cuối cùng, nếu không có keyword nào khớp, finding được gán vào Maintainability (danh mục mặc định).

Bộ phân loại hỗ trợ bốn danh mục Phase 1: Security, Reliability, Maintainability, Performance Efficiency. Bốn danh mục Phase 2 (Functional Suitability, Operability, Compatibility, Transferability) đã được định nghĩa trong code nhưng chưa được kích hoạt.

### 4.3.7. AST Graph Module

Module `core/ast_graph.py` là adapter giao tiếp với GitNexus CLI để xây dựng và truy vấn Knowledge Graph. Module này triển khai một lớp giao tiếp với GitNexus thông qua subprocess, gọi `gitnexus` CLI command để lấy thông tin symbol, context và impact.

Hàm `is_gitnexus_available()` kiểm tra xem GitNexus có được cài đặt và cấu hình không bằng cách chạy `gitnexus status`. Hàm `get_repo_info_for_path()` xác định repository tương ứng với đường dẫn đích. Hàm `resolve_location_to_symbol()` tìm symbol tại một vị trí file và dòng cụ thể. Hàm `get_symbol_context()` trả về thông tin chi tiết về symbol (name, kind, source code, callers). Hàm `get_symbol_impact()` tính blast radius dựa trên đồ thị phụ thuộc.

Nếu GitNexus không khả dụng, module cung cấp MOCK_AST_CONTEXTS để enrichment vẫn hoạt động ở chế độ mock.

### 4.3.8. Scanner Core Module

Module `core/scanner.py` triển khai logic quét với Opengrep. Hàm `run_scan()` xây dựng cấu trúc thư mục rules từ gói opengrep_installer, xác định đường dẫn của opengrep binary, và gọi subprocess với tham số `--json`. Kết quả JSON từ Opengrep được parse và chuyển đổi sang format nội bộ của hệ thống, bao gồm các trường: file, line, column, rule_id, message, severity, CWE ID, OWASP ID.

### 4.3.9. Opengrep Installer

Module `opengrep_installer.py` quản lý việc cài đặt và cập nhật Opengrep binary. Module này kiểm tra phiên bản Opengrep hiện tại, tải binary mới nhất từ GitHub releases nếu cần, và cung cấp đường dẫn đến binary và thư mục rules.
