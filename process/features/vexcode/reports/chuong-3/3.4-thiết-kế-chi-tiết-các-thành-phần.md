# Chương 3: Phân tích và thiết kế hệ thống

## 3.4. Thiết kế chi tiết các thành phần

Phần này mô tả chi tiết từng thành phần trong kiến trúc ba tầng và sáu giai đoạn pipeline, bao gồm cấu trúc file, module chính, và cách chúng kết nối với nhau.

### 3.4.1. Tầng CLI và REST API (Node.js)

Tầng Node.js đóng vai trò là điểm vào duy nhất của hệ thống. Nó cung cấp cả CLI và REST API server, đồng thời là cầu nối với Python engine.

**CLI entry point** (`bin/cli.js`) xử lý ba command chính. Lệnh `scan` nhận tham số `--target`, `--mock-ai`, `--mock-scan`, `--fast` và spawn Python process rồi đọc kết quả từ stdout/file. Lệnh `serve` khởi động Express server trên port do người dùng chỉ định. Lệnh `ui` kết hợp serve và tự động mở trình duyệt.

**Express server** (`src/server.js`) là tầng trung gian duy nhất giữa frontend và engine. Server mount tất cả REST routes, phục vụ static files từ thư mục public (web dashboard đã build), và xử lý CORS. Cấu trúc routes được tổ chức theo resource: config route đọc/ghi cấu hình AI, scan route xử lý lệnh scan và SSE stream, reports route trả về danh sách và nội dung report, apply route thực hiện ghi đề xuất sửa lỗi vào file kèm backup, chat route giao tiếp với AI provider, file route đọc nội dung file gốc, backup route quản lý backup và restore.

**Python bridge** (`src/bridge.js`) là cầu nối giữa Node.js và Python. Bridge dùng `child_process.spawn` để chạy Python script với tham số dòng lệnh. Khi scan, bridge lắng nghe stdout của Python process để parse progress events (JSON struct) và chuyển tiếp thành SSE events đến frontend. Bridge hỗ trợ cancel scan bằng cách kill toàn bộ process tree (Python process + subprocess của nó).

### 3.4.2. Giai đoạn 1 — Scan (Pipeline tầng Python)

Giai đoạn Scan thực hiện phát hiện lỗi bằng ba scanner chạy nối tiếp, kết quả được hợp nhất thành một danh sách findings duy nhất.

**OpenGrep** (scanner chính, `pipeline/scanner.py`): Dùng Semgrep engine để quét toàn bộ cây thư mục target bằng AST pattern matching trên tập luật mặc định từ Semgrep registry. Phát hiện lỗi logic, bảo mật (SQL injection, XSS, command injection), và coding standards. Hỗ trợ chế độ `--fast` để chỉ quét các file đã thay đổi dựa trên git status. Nếu OpenGrep chưa được cài đặt, hệ thống tự cài qua `opengrep_installer.py`.

**Gitleaks** (`pipeline/gitleaks_scanner.py`): Quét riêng secret và credential bị lộ trong toàn bộ dự án bằng so khớp regex và entropy detection. Phát hiện API key, token, mật khẩu bị hardcode và commit nhầm. Kết quả được merge vào cùng danh sách findings với trường `scanner: "gitleaks"`.

**OSV-Scanner** (`pipeline/osv_scanner.py`): Truy vấn lỗ hổng dependency qua CVE database dựa trên file khai báo package (package.json, requirements.txt, Pipfile...). Mỗi finding chứa mã CVE, phiên bản bị ảnh hưởng, và mức độ nghiêm trọng.

Mỗi finding được gán định danh SHA-1 12 ký tự hex, tính từ `file|line|rule_id` (line-based) hoặc `file|rule_id|content_hint` (content-aware) cho Gitleaks và OSV.

**Output:** Danh sách findings thô với file, dòng, rule_id, severity, message, mã CWE, category.

### 3.4.3. Giai đoạn 2 — Enrich (GitNexus Knowledge Graph)

Findings từ Scan chỉ mang thông tin vị trí (file:dòng) và nội dung lỗi, chưa có ngữ cảnh về symbol, callers hay blast radius. Enricher (`pipeline/enricher.py`) giải quyết bằng cách tra cứu GitNexus — knowledge graph của codebase.

Với mỗi finding, enricher xác định symbol tại vị trí `file:dòng`. Nếu tìm thấy, GitNexus bổ sung: tên symbol (hàm/lớp/biến) và mã nguồn xung quanh; danh sách callers — những nơi gọi đến symbol này; blast radius — tập hợp symbol bị ảnh hưởng nếu thay đổi symbol gốc.

Nếu GitNexus chưa được cài đặt hoặc không tìm thấy symbol, giai đoạn này được bỏ qua và findings giữ nguyên dữ liệu gốc — đây là cơ chế graceful degradation.

**Output:** Findings đã có ngữ cảnh AST (symbol, callers, blast radius).

### 3.4.4. Giai đoạn 3 — Internal Analysis

Internal Analysis chạy các phân tích trực tiếp trên source code mà không cần AI, bổ sung thêm findings về chất lượng mã nguồn. Giai đoạn này thực hiện ba phân tích song song:

**Cyclomatic Complexity** (`engine/core/complexity.py`): Dùng Lizard để tính độ phức tạp của từng file — số dòng code (LOC), cyclomatic complexity (CCN), và cognitive complexity. Ngưỡng mặc định: CCN ≤ 25 là LOW, 26–50 là MEDIUM, > 50 là HIGH. Nếu tổng CCN vượt ngưỡng HIGH, hệ thống sinh finding `maintainability.complexity.high-ccn`.

**Cognitive Complexity** (`engine/core/complexity.py`): Phân tích bằng Python AST hoặc heuristic regex (cho ngôn ngữ khác) để đo độ khó hiểu của code. Ngưỡng mặc định 15, sinh finding `maintainability.complexity.high-cognitive`.

**Naming Audit** (`engine/core/naming_audit.py`): Dùng AI để đánh giá chất lượng đặt tên trong code — tên hàm, tên biến, tên lớp. Phát hiện tên mơ hồ, viết tắt khó hiểu, hoặc không nhất quán với style dự án. Các finding này có `rule_id` bắt đầu bằng `maintainability.` và được gắn nhãn `iso_25010: maintainability`.

**Output:** Findings bổ sung về complexity và naming. Toàn bộ findings (từ Scan + Internal Analysis) được chuyển sang giai đoạn AI Resolution.

### 3.4.5. Giai đoạn 4 — AI Resolution (9router API)

Đây là giai đoạn duy nhất gọi đến LLM bên ngoài. AI Resolution nhận toàn bộ findings (từ Scan và Internal Analysis) và xử lý qua ba stage liên tiếp:

**Smart skip:** Trước khi vào stage Analyze, hệ thống kiểm tra severity và confidence của từng finding. Chỉ những finding có severity từ WARNING trở lên và confidence == HIGH mới được gửi đến AI — những lỗi gần như chắc chắn là thật, cần đánh giá sâu. Findings còn lại bị bỏ qua, không tốn token.

**Cache:** Với cùng tổ hợp `rule_id` và đoạn code báo lỗi giống hệt, hệ thống lưu kết quả từ lần gọi AI đầu tiên vào file JSON cục bộ (`~/.vexcode/ai_cache/`) và tái sử dụng. Cache hoạt động theo exact match.

**Stage 1 — Analyze** (`engine/core/ai_resolver.py`): Mỗi finding vượt qua smart skip được gửi kèm ngữ cảnh AST (source code, callers, blast radius, mã nguồn xung quanh). Model AI phân loại thành `confirmed` (lỗi thật, cần sinh mã sửa), `hotspot` (cần xem xét thêm), hoặc `false_positive` (báo động sai). Findings `hotspot` và `false_positive` dừng tại đây.

**Stage 2 — Fix**: Findings `confirmed` được gửi kèm ngữ cảnh mở rộng — source code callers, blast radius, surrounding code, và kết quả semantic search từ GitNexus — đến model AI riêng (thường là model mạnh nhất). Model trả về mã sửa cụ thể (`remediation_code`) và giải thích (`suggestion`). Mã sửa được sanitize trước khi lưu (loại bỏ markdown fence, kiểm tra ký tự đặc biệt).

**Stage 3 — Review**: Model AI thứ ba (cấu hình cân bằng chi phí-chất lượng) nhận kết quả từ stage Fix để đánh giá. Nếu `approved`, resolution giữ nguyên. Nếu `rejected`, model review cung cấp bản sửa thay thế kèm `review_comment`.

**Output:** Resolution cho mỗi rule — gồm classification, suggestion, remediation_code, review_decision, review_comment. Resolutions được lưu trong `report.ai_resolutions` với key là `rule_id`.

### 3.4.6. Giai đoạn 5 — Report & Quality Gate

Report & Quality Gate là lớp tổng hợp kết quả và đánh giá chất lượng tổng thể. Giai đoạn này không can thiệp vào pipeline xử lý chính.

**Assemble report** (`pipeline/reporter.py`): Hệ thống gộp findings, resolutions, metrics, git state và metadata thành một file JSON duy nhất tại `~/.vexcode/reports/{projectName}/{timestamp}.json`. Nếu cờ `--format md` được bật, hệ thống xuất thêm bản Markdown cho người đọc. Nếu không có `--no-sarif`, hệ thống xuất song song file SARIF 2.1.0 sidecar phục vụ tích hợp CI/CD.

**Threshold evaluation (Quality Gate)** (`pipeline/thresholds.py`): Nếu người dùng cung cấp file TOML qua `--thresholds`, hệ thống đánh giá findings dựa trên các ngưỡng cấu hình. Kết quả (passed/failed + danh sách vi phạm) được ghi vào report. CLI flag `--fail-on-threshold` trả về exit code 1 khi quality gate không đạt.

**Output:** Report JSON (kèm SARIF nếu không dùng `--no-sarif`). Threshold violations nếu có.

### 3.4.7. Giai đoạn 6 — View & Act (Web Dashboard + Express API)

Tầng giao diện là Single Page Application được xây dựng với React 19 và TypeScript, phục vụ bởi Express server qua REST API.

**Dashboard** (`packages/web/`): Hiển thị tổng quan dự án — phân bố severity, top file có nhiều lỗi nhất, điểm sức khỏe dự án. Dashboard tự động cập nhật sau mỗi lần scan hoàn tất.

**Issues page**: Danh sách findings dạng bảng với các thao tác lọc theo severity/category/status, tìm kiếm toàn văn, sắp xếp và phân trang.

**Code Inspector**: Component chính để xem và thao tác với findings. Inspector hiển thị mã nguồn với dòng lỗi được highlight trong CodeMirror, side panel hiển thị đề xuất sửa lỗi kèm diff trực quan, và cho phép chat với AI về finding cụ thể. Inspector cũng hiển thị AST context (callers, callees, blast radius) nếu GitNexus được cài đặt.

**Settings Drawer**: Người dùng chọn AI provider (OpenAI, Anthropic, Google, 9router, NVIDIA), nhập API key, chọn model. Thay đổi có hiệu lực ngay lập tức, không cần restart server.

**Vòng đời finding trên giao diện**: Mỗi finding có bốn trạng thái — `open` (mặc định), `applied` (đã áp dụng bản sửa), `false_positive` (báo sai), `ignored` (bỏ qua). Trạng thái được lưu trong file báo cáo JSON và tồn tại qua các lần tải lại trang. Trước khi apply, hệ thống tự động tạo backup tại `~/.vexcode/backups/`. Người dùng có thể rollback từ backup bất cứ lúc nào.

### 3.4.8. Hệ thống AI Provider

Hệ thống hỗ trợ nhiều nhà cung cấp AI qua một interface thống nhất (`engine/config/ai_config.py`). Cấu hình provider được đọc từ file `.env` mỗi lần scan. Khi người dùng thay đổi provider trong Settings Drawer, cấu hình được ghi vào file `.env` và API sẽ dùng provider mới cho lần scan tiếp theo.

Cơ chế fallback: nếu không có API key, hệ thống trả về mock resolution kèm thông báo. Nếu base URL hoặc provider chưa được cấu hình, tương tự. Nếu flag `--mock-ai` được bật, tất cả resolutions đều là mock — phục vụ kiểm thử không tốn chi phí API.

### 3.4.9. REST API

Tất cả giao tiếp giữa web dashboard và server đều qua REST API với JSON:

| Resource | Phương thức | Endpoint | Mô tả |
|----------|------------|---------|-------|
| Config | GET/PUT | `/api/config` | Đọc và ghi cấu hình AI provider |
| Scan | GET | `/api/scan/stream` | SSE cho real-time progress |
| Scan | POST | `/api/scan` | Scan đồng bộ |
| Scan | POST | `/api/scan/cancel` | Hủy scan đang chạy |
| Scan | POST | `/api/scan/refresh-ai` | Chạy lại AI resolution trên báo cáo cũ |
| Reports | GET | `/api/reports` | Danh sách project |
| Reports | GET | `/api/reports/:project` | Danh sách report của project |
| Reports | GET | `/api/report/:project/:id` | Nội dung một report |
| Apply | POST | `/api/apply` | Áp dụng bản sửa vào file kèm backup |
| Chat | POST | `/api/chat` | Hỏi AI về một finding cụ thể |
| Files | GET | `/api/file-content` | Đọc nội dung file |
| Backup | GET | `/api/backup` | Liệt kê backup |
| Backup | POST | `/api/backup` | Restore file từ backup |
