# Chương 3: Phân tích và thiết kế hệ thống

## 3.4. Thiết kế chi tiết các thành phần

### 3.4.1. Tầng CLI và REST API (Node.js)

**CLI entry point** (`bin/cli.js`) xử lý ba command chính. `vexcode analyze` nhận tham số target path, fast mode flag, và mock flag. Lệnh này spawn Python process và đọc kết quả từ stdout/file. `vexcode serve` khởi động Express server trên port do người dùng chỉ định, phục vụ web dashboard và REST API. `vexcode ui` kết hợp cả serve và tự động mở trình duyệt.

Phía Python engine hỗ trợ thêm các flag mở rộng: `--no-sarif` bỏ qua xuất SARIF sidecar, `--format md` xuất báo cáo Markdown, `--thresholds` chỉ định file TOML quality gate, `--explain` in giải thích findings ra stdout, `--refresh-ai` chạy lại AI resolution trên báo cáo có sẵn mà không cần quét lại, và `--fail-on-threshold` thoát mã lỗi nếu quality gate không đạt.

**Express server** (`src/server.js`) là tầng trung gian duy nhất giữa frontend và engine. Server đảm nhận các nhiệm vụ: mount tất cả REST routes, phục vụ static files từ thư mục public (web dashboard đã build), và xử lý CORS. Cấu trúc routes được tổ chức theo từng file riêng: config route đọc/ghi cấu hình AI, scan route xử lý lệnh scan và SSE stream, reports route trả về danh sách và nội dung report, apply route thực hiện ghi đề xuất sửa lỗi vào file kèm backup.

**Python bridge** (`src/bridge.js`) là cầu nối giữa Node.js và Python. Bridge sử dụng child_process.spawn để chạy Python script với tham số dòng lệnh. Khi scan, bridge lắng nghe stdout của Python process để parse progress events và chuyển tiếp thành SSE events đến frontend. Bridge cũng hỗ trợ cancel scan bằng cách kill process tree.

### 3.4.2. Tầng phân tích (Python)

Tầng phân tích là pipeline tám giai đoạn chạy tuần tự, mỗi giai đoạn nhận đầu ra của giai đoạn trước và bổ sung thông tin. Pipeline được tổ chức thành ba nhóm: quét (scan), xử lý (enrich → complexity → dedup → classify), và giải trình (naming_audit → ai_resolve → report).

**Giai đoạn 1 — Scanner.** (`pipeline/scanner.py`, `pipeline/gitleaks_scanner.py`, `pipeline/osv_scanner.py`) Scanner thực chất là ba sub-scanner chạy nối tiếp:

- **OpenGrep** (chính): Quét tĩnh mã nguồn với bộ luật phát hiện lỗi bảo mật, SQL injection, XSS, hardcoded secrets, deprecated API. Nếu chế độ fast được bật, chỉ quét các file đã thay đổi dựa trên git status.
- **Gitleaks** (bổ sung): Quét riêng secret và credential trong toàn bộ dự án, phát hiện các key/token bị commit nhầm. Kết quả được merge vào cùng danh sách findings.
- **OSV-Scanner** (bổ sung): Quét lỗ hổng thư viện dependency dựa trên file khai báo package (package.json, requirements.txt). Mỗi finding chứa mã CVE và thông tin phiên bản bị ảnh hưởng.

Kết quả sau giai đoạn này là danh sách findings thô với file, dòng, rule_id, CWE ID và mô tả lỗi.

**Giai đoạn 2 — Enricher.** (`pipeline/enricher.py`) Làm giàu findings với ngữ cảnh từ GitNexus Knowledge Graph. Với mỗi finding, enricher xác định symbol liên quan (tên hàm, lớp), danh sách caller, phạm vi ảnh hưởng và mức độ ảnh hưởng trong kiến trúc. Nếu GitNexus chưa được cài đặt, giai đoạn này được bỏ qua và findings giữ nguyên dữ liệu gốc.

**Giai đoạn 3 — Complexity.** (`engine/core/complexity.py`) Tính độ phức tạp mã nguồn cho từng file bằng Lizard: số dòng (LOC), độ phức tạp cyclomatic và độ phức tạp nhận thức (cognitive complexity). Các ngưỡng cảnh báo có thể được cấu hình qua CLI (`--ccn-threshold`, `--cognitive-threshold`).

**Giai đoạn 4 — Dedup.** (`engine/core/dedup.py`) Loại bỏ các findings trùng lặp phát sinh khi nhiều scanner cùng phát hiện một lỗi. Dedup so khớp theo rule_id, file path, dòng và nội dung thông báo. Số dòng và token tối thiểu để phát hiện trùng lặp có thể điều chỉnh (`--dup-min-lines`, `--dup-min-tokens`).

**Giai đoạn 5 — Classify.** Phân loại từng finding vào một trong bốn nhóm chất lượng theo ISO/IEC 25010: Security, Reliability, Maintainability và Performance Efficiency. Việc phân loại dựa trên rule_id, CWE ID và nội dung thông báo lỗi.

**Giai đoạn 6 — Naming Audit.** (`engine/core/`) Kiểm tra tên biến, hàm, lớp có khó hiểu hay không bằng AI audit. Các tên viết tắt, tên chung chung hoặc không tuân theo naming convention được ghi nhận để đề xuất cải tiến.

**Giai đoạn 7 — AI Resolve.** (`engine/core/ai_resolver.py`) Gọi mô hình ngôn ngữ lớn cho từng finding để tạo đề xuất sửa lỗi cụ thể. Mỗi resolution gồm explanation, code diff và mức độ rủi ro (risk assessment). AI resolution chạy song song với tối đa 3 worker. Nếu chưa cấu hình AI, resolver trả về mock resolutions. Flag `--mock-ai` buộc dùng mock cho tất cả. Engine cũng hỗ trợ chạy lại AI resolution trên báo cáo cũ qua `--refresh-ai` mà không cần quét lại.

**Giai đoạn 8 — Reporter.** (`pipeline/reporter.py`, `pipeline/sarif_builder.py`) Tổng hợp toàn bộ dữ liệu thành báo cáo có cấu trúc. Engine xuất đồng thời ba định dạng:

- **JSON:** Báo cáo chính (VexCode format) gồm metadata, findings đã làm giàu, resolutions, metrics, naming issues. Ghi vào `~/.vexcode/reports/{projectName}/{timestamp}.json`.
- **SARIF:** Sidecar dạng SARIF 2.1.0 cho tích hợp với CI/CD và các công cụ hỗ trợ SARIF.
- **Markdown:** Báo cáo người đọc được nếu dùng flag `--format md`.

Kết quả quality gate (nếu cấu hình file TOML với `--thresholds`) cũng được đính kèm vào báo cáo.

### 3.4.3. Tầng giao diện (React)

Tầng giao diện là Single Page Application (SPA) được xây dựng với React 19 và TypeScript. Giao diện được chia làm các khu vực chính.

**Dashboard** hiển thị tổng quan dự án: phân bố mức độ nghiêm trọng của findings, top file có nhiều lỗi nhất, và các chỉ số thống kê (tổng số findings, số lỗi security, số lỗi maintainability). Dashboard tự động cập nhật sau mỗi lần scan hoàn tất.

**Issues page** cung cấp danh sách findings dạng bảng có thể sắp xếp và lọc theo mức độ nghiêm trọng, loại lỗi, ngôn ngữ và từ khóa tìm kiếm. Khi chọn một finding, người dùng thấy mã nguồn tại vị trí lỗi, đề xuất sửa lỗi từ AI, và các nút để áp dụng hoặc hỏi AI thêm.

**Code Inspector** là component chính để xem và thao tác với findings. Inspector hiển thị mã nguồn với dòng lỗi được highlight, side panel hiển thị đề xuất sửa lỗi và cho phép chat với AI về finding cụ thể. Inspector cũng hiển thị AST context (callers, callees, blast radius) nếu GitNexus được cài đặt.

**Scan Config Panel** cho phép người dùng chọn đường dẫn dự án, chọn chế độ quét (fast hoặc full) và bắt đầu scan.

**Scan Progress** hiển thị tiến độ quét theo từng bước (scan, enrich, complexity, dedup, classify, naming audit, AI resolve, report). Mỗi bước có trạng thái pending, active hoặc completed.

**Settings Drawer** cho phép người dùng chọn AI provider, nhập API key, base URL và tên model. Thay đổi có hiệu lực ngay lập tức, không cần restart server.

### 3.4.4. Hệ thống AI Provider

Hệ thống hỗ trợ nhiều nhà cung cấp AI thông qua một interface thống nhất. Cấu hình provider được đọc từ file .env mỗi lần scan. Khi người dùng thay đổi provider trong Settings Drawer, cấu hình được ghi vào file .env và API sẽ gọi provider mới cho lần scan tiếp theo.

Cơ chế fallback của AI provider hoạt động như sau: nếu không có API key, hệ thống trả về thông báo "API key is required" kèm mock resolution. Nếu base URL hoặc provider chưa được cấu hình, hệ thống trả về thông báo "AI config is incomplete" kèm mock resolution. Nếu flag --mock-ai được bật, tất cả resolutions đều là mock.

### 3.4.5. REST API

Tất cả giao tiếp giữa web dashboard và server đều qua REST API với JSON. Các API endpoint được tổ chức theo resource:

- Config: GET/PUT `/api/config` để đọc và ghi cấu hình AI.
- Scan: GET `/api/scan/stream` với SSE cho real-time progress, POST `/api/scan` cho scan đồng bộ, POST `/api/scan/cancel` để hủy, POST `/api/scan/refresh-ai` chạy lại AI resolution trên báo cáo cũ mà không quét lại.
- Reports: GET `/api/reports` lấy danh sách project, GET `/api/reports/:project` lấy danh sách report, GET `/api/report/:project/:id` lấy nội dung report.
- Apply: POST `/api/apply` áp dụng bản sửa vào file kèm backup.
- Chat: POST `/api/chat` hỏi AI thêm về finding.
- Files: GET `/api/file-content` đọc nội dung file.
- Backup: GET `/api/backup` liệt kê backup, POST `/api/backup` restore.
