# Chương 3: Phân tích và thiết kế hệ thống

## 3.2. Xác định yêu cầu hệ thống

Hệ thống được xây dựng dựa trên những tính năng cốt lõi. Các tính năng này là cơ sở để thiết kế kiến trúc tổng thể và pipeline xử lý ở những phần sau.

### 3.2.1. Yêu cầu chức năng

*Bảng 3.5. Danh sách yêu cầu chức năng*

| Mã | Chức năng | Mô tả |
|----|----------|-------|
| F01 | Quét mã nguồn | Người dùng chọn thư mục dự án và bắt đầu quét. Hệ thống thực thi pipeline phân tích, hiển thị tiến trình theo thời gian thực. Có thể hủy quét giữa chừng. |
| F02 | Xem Dashboard | Dashboard hiển thị tổng quan kết quả: tổng số lỗi, phân bố theo mức độ nghiêm trọng (severity), phân bố theo thể loại, top file có nhiều lỗi nhất, điểm sức khỏe dự án. |
| F03 | Xem danh sách lỗi | Người dùng duyệt danh sách lỗi đã được phân loại, cho phép lọc theo severity/category/status, tìm kiếm toàn văn bản, sắp xếp và phân trang. |
| F04 | Xem chi tiết lỗi | Click vào một lỗi để xem vị trí (file, dòng) được highlight trong CodeMirror, phân tích nguyên nhân từ AI, đề xuất sửa chữa kèm so sánh diff. |
| F05 | Ask AI | Người dùng có thể hội thoại với AI về ngữ cảnh của lỗi để được giải thích thêm hoặc tư vấn hướng xử lý. |
| F06 | Áp dụng sửa lỗi | Người dùng có thể áp dụng đề xuất sửa chữa từ AI vào file mã nguồn. Hệ thống tự động backup trước khi sửa và hỗ trợ rollback nếu cần. |
| F07 | Đánh dấu trạng thái | Người dùng đánh giá từng lỗi với các trạng thái: open (cần xem lại), applied (đã sửa), false_positive (báo sai), ignored (bỏ qua). |
| F08 | Xuất báo cáo | Người dùng xuất kết quả phân tích dưới dạng SARIF 2.1.0 hoặc JSON. |
| F09 | Cấu hình AI | Người dùng chọn nhà cung cấp AI (OpenAI, Anthropic, Google, 9router, NVIDIA), nhập API key, chọn model. |

### 3.2.2. Yêu cầu phi chức năng

*Bảng 3.6. Danh sách yêu cầu phi chức năng*

| Mã | Yêu cầu | Mô tả | Cách kiểm chứng |
|----|---------|-------|------------------|
| NF01 | Hiệu suất | Quét và phân tích tĩnh dưới 10.000 file trong không quá 5 phút (không tính thời gian AI xử lý). | Chạy `vexcode scan --target <dir> --mock-ai`, ghi thời gian bằng `time` command. |
| NF02 | AI Resolution | Pipeline 3 tầng (analyze → fix → review), chỉ gọi AI cho lỗi nghiêm trọng, còn lại dùng mock. Timeout 90s, retry 2 lần. | Chạy scan với `--mock-ai` để xác nhận không có AI call. Chạy scan thật, kiểm tra pipeline metrics (cache hits, skip count, AI calls) trong log. |
| NF03 | Khả năng fallback | Mỗi tầng độc lập — thiếu GitNexus thì bỏ qua enrichment, thiếu API key thì dùng mock, Opengrep lỗi thì dùng mock scan. | Lần lượt vô hiệu hóa GitNexus (xóa khỏi PATH), bỏ API key, kiểm tra kết quả scan không bị rỗng và có log cảnh báo. |
| NF04 | An toàn dữ liệu | Backup trước khi sửa, chống path traversal bằng workspace boundary. | Apply một fix trên dashboard, kiểm tra file `.backup` được tạo. Gửi path traversal payload (`../../etc/passwd`) vào API, xác nhận bị chặn với mã 400. |
| NF05 | Cấu hình động | Đổi AI provider khi đang chạy, không cần restart. | Đổi từ OpenAI sang Anthropic trong Settings Drawer khi server đang chạy, chạy scan lại, kiểm tra log ghi nhận provider mới. |
