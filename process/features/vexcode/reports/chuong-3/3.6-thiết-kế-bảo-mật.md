# Chương 3: Phân tích và thiết kế hệ thống

## 3.6. Thiết kế bảo mật

### 3.6.1. Mô hình mối đe dọa (Threat Model)

Hệ thống được phân tích theo khung STRIDE, tập trung vào các tài sản cần bảo vệ, các mối đe dọa tiềm ẩn, và biện pháp giảm thiểu tương ứng.

**Tài sản cần bảo vệ (Assets):**

| Tài sản | Mô tả | Mức độ rủi ro |
|---------|-------|-------------|
| API key AI provider | Lưu trong file `.env`, dùng để gọi OpenAI/Anthropic/Google API | Rất cao — nếu lộ, kẻ tấn công có thể sử dụng tài khoản AI của nạn nhân |
| File mã nguồn người dùng | Đường dẫn target mà hệ thống quét | Cao — hệ thống đọc và có thể ghi đè file |
| Báo cáo JSON | Chứa findings, resolutions, đường dẫn file, nội dung code lỗi | Trung bình — có thể chứa thông tin nhạy cảm |
| Backup file | Bản sao mã nguồn trước khi apply fix | Cao — cần bảo vệ khỏi truy cập trái phép |

**Các mối đe dọa theo STRIDE:**

| Loại | Mối đe dọa | Kịch bản | Biện pháp giảm thiểu |
|------|-----------|---------|---------------------|
| **Spoofing** | Giả mạo API key | Kẻ tấn công đánh cắp API key từ file `.env` do cấu hình sai quyền truy cập | Lưu trong `.env` (không commit git), chmod 600 |
| **Tampering** | Path traversal | Kẻ tấn công gửi đường dẫn `../../etc/passwd` để đọc file hệ thống | `isPathSafe()` kiểm tra workspace boundary, chặn `..` |
| **Tampering** | Ghi đè file tùy ý | Lợi dụng lỗ hổng path traversal để ghi đè file nhạy cảm | Chỉ ghi vào file trong thư mục target đã kiểm tra; backup trước mọi thay đổi |
| **Tampering** | Thay đổi báo cáo | Chỉnh sửa file JSON report để ẩn lỗi thật | Reports nằm trong thư mục user home, không bị ảnh hưởng bởi target scan |
| **Repudiation** | Phủ nhận hành động | Developer phủ nhận đã apply một bản sửa | Mọi hành động apply được log, backup được lưu với timestamp |
| **Information Disclosure** | Lộ đường dẫn nội bộ | Error message trả về đường dẫn tuyệt đối đầy đủ | Sanitize error message trước khi trả về client |
| **Information Disclosure** | Lộ nội dung code trong report | Báo cáo chứa đoạn code lỗi có thể nhạy cảm | Report lưu trong `~/.vexcode/`, không tự động gửi đi đâu |
| **Denial of Service** | Quét thư mục quá lớn | Kẻ tấn công chỉ định thư mục rất lớn làm hệ thống treo | Giới hạn số file có thể quét, timeout cho từng stage |
| **Elevation of Privilege** | Lợi dụng API để thực thi code | Gửi request tùy ý đến `/api/apply` để ghi đè file | Chỉ cho phép apply vào file trong thư mục target đã kiểm tra |

### 3.6.2. Bề mặt tấn công (Attack Surface)

**REST API endpoints** — Tất cả endpoint đều nhận input từ người dùng:

- `POST /api/scan`: Nhận `target_path` — kẻ tấn công có thể thử path traversal.
- `POST /api/apply`: Nhận `file_path` + `remediation_code` — kẻ tấn công có thể thử ghi đè file ngoài target.
- `GET /api/file-content`: Nhận `path` — kẻ tấn công có thể thử đọc file ngoài workspace.
- `GET /api/backup`: Nhận `project`, `filename` — kẻ tấn công có thể thử truy cập backup tùy ý.

**AI Provider Communication** — API key được gửi qua header `Authorization` đến các provider bên ngoài (OpenAI, Anthropic...). Nếu kẻ tấn công có thể sniff network traffic trên máy người dùng, có thể đánh cắp API key. Rủi ro này thuộc về môi trường mạng của người dùng, không thuộc phạm vi kiểm soát của hệ thống. Hệ thống chỉ lưu trữ API key trong file `.env` cục bộ, không gửi qua mạng trừ khi gọi trực tiếp đến AI provider.

**File System** — Hệ thống đọc và ghi file trên đĩa. Đây là bề mặt tấn công lớn nhất: bất kỳ lỗi nào trong việc kiểm tra đường dẫn đều có thể cho phép kẻ tấn công đọc/ghi file tùy ý.

### 3.6.3. Các cơ chế bảo mật

**Path Safety — Chống path traversal:**

Hàm `isPathSafe()` (`packages/cli/src/utils.js`) kiểm tra mọi đường dẫn đầu vào trước khi sử dụng. Kiểm tra case-insensitive prefix: đường dẫn phải bắt đầu bằng workspace boundary đã xác nhận. Mọi đường dẫn chứa `..` đều bị chặn. Kiểm tra này áp dụng cho: target path khi scan, file path khi đọc nội dung, file path khi apply fix, đường dẫn backup khi restore.

```javascript
// Ví dụ kiểm tra
function isPathSafe(requestedPath, workspaceRoot) {
  const resolved = path.resolve(requestedPath);
  const rootResolved = path.resolve(workspaceRoot);
  return resolved.startsWith(rootResolved);
}
```

**Backup trước khi sửa:**

Trước khi ghi bất kỳ bản sửa nào vào file, hệ thống tự động tạo một bản sao tại `~/.vexcode/backups/{projectName}/{timestamp}_{filename}`. Người dùng có thể restore file gốc từ backup bất cứ lúc nào qua API. Backup được lưu trong thư mục user home — tách biệt hoàn toàn khỏi thư mục dự án đang quét — nên không bị ảnh hưở ngay cả khi kẻ tấn công kiểm soát được target path.

**Workspace Boundary:**

Khi scan, hệ thống xác định workspace boundary bằng cách tìm thư mục gốc chứa file cấu hình (`.git`, `package.json`, `pyproject.toml`) và chỉ quét trong phạm vi đó. Điều này ngăn chặn việc scan các thư mục hệ thống như `/etc/`, `~/.ssh/`, hoặc các thư mục outside the project.

**API Key Protection:**

API key AI provider được lưu trong file `.env` tại `packages/engine/`. File này nằm trong `.gitignore`, không bao giờ được commit lên git. Khi scan, key chỉ được đọc bởi Python engine trong cùng máy, không được gửi qua mạng. Người dùng có trách nhiệm bảo vệ quyền truy cập file `.env` (chmod 600 trên Linux/macOS).

**Input Sanitization cho AI Provider:**

Trước khi gửi code đến AI provider, hệ thống sanitize nội dung: loại bỏ markdown fence (```), kiểm tra ký tự đặc biệt có thể gây prompt injection, giới hạn độ dài nội dung gửi đi (`MAX_CODE_CHARS`). Điều này giảm nguy cơ prompt injection qua mã nguồn độc hại.

**CORS và API Authentication:**

Express server chỉ phục vụ web dashboard từ cùng một origin trong chế độ mặc định. CORS được cấu hình restrictive — chỉ cho phép origin của web dashboard. Hiện tại hệ thống không có authentication riêng (user/pass, JWT) vì được thiết kế cho usage cục bộ trên máy developer. Nếu triển khai remote, cần bổ sung authentication middleware.

### 3.6.4. Hạn chế bảo mật hiện tại

Hệ thống có một số hạn chế cần lưu ý:

**Không có authentication cho REST API** — Bất kỳ ai có quyền truy cập port của Express server đều có thể gọi mọi endpoint, bao gồm apply fix tùy ý. Trong môi trường dùng chung (shared server), đây là rủi ro. Giải pháp tạm thời: chỉ bind `localhost` thay vì `0.0.0.0`.

**Không có rate limiting** — Không giới hạn số lần gọi API, cho phép brute force hoặc DoS cục bộ.

**Backup không mã hóa** — File backup được lưu dạng plain text, ai có quyền truy cập thư mục `~/.vexcode/backups/` đều đọc được.

**API key trong biến môi trường của process** — Trên một số hệ điều hành, biến môi trường có thể bị đọc bởi process khác cùng user. Consider dùng secret manager (Vault, AWS Secrets Manager) cho production.

### 3.6.5. Checklist bảo mật triển khai

| Biện pháp | Trạng thái | Ghi chú |
|-----------|-----------|---------|
| File `.env` trong `.gitignore` | ✓ Đã cấu hình | API key không bị commit |
| `chmod 600` cho `.env` | Người dùng tự thực hiện | Không thể kiểm soát tự động |
| Path safety check trước mọi file I/O | ✓ Có trong code | Kiểm tra workspace boundary |
| Backup trước khi apply | ✓ Có trong code | Tự động, không cần thao tác |
| CORS restrictive | ✓ Đã cấu hình | Chỉ localhost origin |
| Sanitize code trước khi gửi AI | ✓ Có trong code | Loại bỏ markdown fence |
| Không log API key | ✓ Đã cấu hình | Logger lọc Authorization header |
| Input validation cho SSE endpoint | Cần kiểm tra | Người dùng gửi target_path có thể qua lại |
| Bind server `localhost` thay vì `0.0.0.0` | Khuyến nghị | Quan trọng nếu dùng trong mạng chia sẻ |
