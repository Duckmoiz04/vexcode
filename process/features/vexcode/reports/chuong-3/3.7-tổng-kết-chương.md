# Chương 3: Phân tích và thiết kế hệ thống

## 3.7. Tổng kết chương

Chương này đã trình bày toàn bộ quá trình phân tích và thiết kế cho hệ thống Vexcode — từ yêu cầu, kiến trúc, thiết kế chi tiết, đến bảo mật.

**Về yêu cầu hệ thống (mục 3.2):** Chương xác định chín yêu cầu chức năng (F01–F09) phủ toàn bộ vòng đời sử dụng: từ scan, xem dashboard, duyệt danh sách lỗi, xem chi tiết và diff, hỏi AI, apply fix, đánh dấu trạng thái, xuất báo cáo, đến cấu hình AI provider. Năm yêu cầu phi chức năng (NF01–NF05) đảm bảo hệ thống đạt tiêu chuẩn về hiệu suất (quét dưới 5 phút cho 10.000 file), AI resolution 3 tầng với smart skip và cache, graceful fallback ở mọi tầng, an toàn dữ liệu (backup, path safety), và cấu hình động không cần restart.

**Về kiến trúc tổng thể (mục 3.3):** Hệ thống sử dụng kiến trúc ba tầng (Node.js / Python / React) với ba nguyên tắc thiết kế: hybrid SAST-first để tối ưu chi phí AI, graceful degradation ở mọi tầng để đảm bảo luôn hoạt động, và tách biệt ba nền tảng để dễ bảo trì. Luồng dữ liệu chính đi từ người dùng qua Node.js, sang Python pipeline, rồi trả về dashboard — mỗi giai đoạn đều ghi progress để hiển thị real-time.

**Về thiết kế chi tiết (mục 3.4):** Sáu giai đoạn pipeline được thiết kế theo nguyên tắc mỗi giai đoạn chỉ bổ sung thông tin, không làm lại việc của giai đoạn khác. Scan dùng ba scanner (OpenGrep, Gitleaks, OSV) hợp nhất kết quả. Enrich bổ sung ngữ cảnh AST từ GitNexus. Internal Analysis chạy song song complexity và naming audit. AI Resolution xử lý ba stage (Analyze → Fix → Review) với smart skip và cache để tối ưu token. Report & Quality Gate tổng hợp kết quả ra JSON, SARIF, Markdown. View & Act cung cấp giao diện trực quan với diff view và rollback.

**Về thiết kế dữ liệu (mục 3.5):** Hệ thống dùng file JSON thay vì database — đơn giản hóa triển khai, dễ backup, và không cần schema migration. Finding là thực thể trung tâm với 37 trường bao phủ toàn bộ vòng đời lỗi. Định danh SHA-1 đảm bảo cùng lỗi qua các lần quét luôn có cùng ID, cho phép so sánh để phát hiện lỗi mới, đã sửa, và tái phát.

**Về bảo mật (mục 3.6):** Phân tích STRIDE xác định các mối đe dọa chính: path traversal (cao nhất), đánh cắp API key, và sửa đổi file trái phép. Các cơ chế bảo mật đã triển khai bao gồm: kiểm tra workspace boundary trước mọi file I/O, backup tự động trước khi apply, sanitize code trước khi gửi AI, CORS restrictive, và log lọc Authorization header. Các hạn chế cần lưu ý gồm: chưa có authentication cho REST API, backup chưa mã hóa, và khuyến nghị bind localhost khi dùng trong mạng chia sẻ.

**Về pipeline tổng hợp:** Sáu giai đoạn kết nối thành pipeline hoàn chỉnh, mỗi giai đoạn đều có cơ chế fallback riêng. Finding có vòng đời bốn trạng thái (open → applied / false_positive / ignored) được quản lý trong chính file báo cáo JSON. Kiến trúc này đảm bảo hệ thống luôn trả kết quả hữu ích, ngay cả khi một số thành phần gặp sự cố.
