## 3.2. Thiết kế kiến trúc tổng thể

### 3.2.1. Nguyên tắc thiết kế

Hệ thống được xây dựng trên ba nguyên tắc chính:

**Hybrid SAST-first, AI hỗ trợ.** Hệ thống dùng SAST (Static Application Security Testing) làm lớp phát hiện lỗi chính, AI (multi-provider: OpenAI, Anthropic, Google, 9router) chỉ đóng vai trò đánh giá và đề xuất sửa. Cách tiếp cận này tận dụng độ chính xác của SAST cho các lỗi đã biết (pattern-based), đồng thời dùng AI để giải quyết false positive và cung cấp mã sửa. Việc phát hiện bằng SAST trước còn giảm đáng kể số lượng token cần gửi đến AI, yếu tố quyết định chi phí và thời gian.

**Graceful degradation ở mọi tầng.** Mỗi thành phần phụ thuộc (GitNexus, AI provider, scanner) đều có cơ chế fallback độc lập. Thiếu GitNexus thì bỏ qua enrichment; thiếu AI thì dùng mock resolution; scanner lỗi thì dùng mock scan. Hệ thống luôn trả về kết quả, không fail hoàn toàn.

**Tách rõ vai trò theo nền tảng.** Node.js lo CLI và API server; Python chạy pipeline phân tích; React hiển thị giao diện. Mỗi phần độc lập, chỉ giao tiếp với nhau qua JSON.

### 3.2.2. Sơ đồ kiến trúc ba tầng

Hệ thống được chia thành ba phần chính, mỗi phần chạy trên nền tảng khác nhau và có trách nhiệm riêng:

**Tầng giao diện (React SPA):** Chạy trên trình duyệt của người dùng. Bao gồm các trang Dashboard, Issues, Code Inspector, và Settings. Tầng này chỉ làm một việc: hiển thị thông tin và gửi yêu cầu đến server.

**Tầng API (Node.js - Express):** Là "cầu nối" giữa giao diện và phần phân tích. Xử lý các lệnh từ CLI (scan, serve, ui), nhận yêu cầu REST API từ frontend, kiểm tra xác thực và giới hạn tốc độ, đồng thời điều phối tiến trình Python.

**Tầng phân tích (Python):** Thực hiện toàn bộ việc quét và xử lý mã nguồn. Gồm 5 giai đoạn: Scan, Dedup, Enrich, Resolve, và Report. Tầng này không biết về web hay Express, chỉ đọc cấu hình và ghi kết quả ra file.

Cách hai tầng trên giao tiếp với tầng Python:
- Node.js "khởi chạy" tiến trình Python và giao tiếp qua đầu vào/ra chuẩn (stdio) dùng JSON.
- Node.js lấy kết quả từ Python rồi chuyển tiếp cho frontend qua REST API hoặc SSE (cập nhật tiến độ thời gian thực).

Lựa chọn ba tầng riêng biệt vì: Python có hệ sinh thái phong phú cho phân tích tĩnh (AST parsing, complexity metrics, subprocess management); Node.js xử lý I/O bất đồng bộ tốt cho API server; React phù hợp cho dashboard tương tác với real-time progress.

### 3.2.3. Pipeline phân tích

Mỗi phiên quét đi qua bốn giai đoạn tuần tự. Bước Dedup được thực hiện ngay sau Scan, nộ

| Giai đoạn | Module | Chức năng chính |
|-----------|--------|-----------------|
| **Scan** | OpenGrep, Gitleaks, OSV-Scanner + Core dedup | Phát hiện lỗi bảo mật, quét secret, phân tích dependency. Dedup loại trùng ngay sau khi hợp nhất kết quả. |
| **Enrich** | GitNexus (Knowledge Graph) | Bổ sung symbol, callers, blast radius qua AST context |
| **Resolve** | Complexity + Naming Audit + AI Resolution | Phân tích độ phức tạp, kiểm tra naming, phân loại confirmed/hotspot/false_positive, sinh mã sửa, review (3 stage) |
| **Report & Quality Gate** | Reporter + Thresholds | Tổng hợp JSON/SARIF/MD, kiểm tra threshold, trả về kết quả |

Sau khi pipeline sinh báo cáo, người dùng xem findings, xem diff và apply/rollback qua giao diện.

Mỗi giai đoạn đều có cơ chế graceful degradation độc lập:
- **Scan**: nếu OpenGrep/Gitleaks/OSV không khả dụng, pipeline chuyển sang `--mock-scan` (findings giả lập) để các giai đoạn sau vẫn chạy thử.
- **Enrich**: nếu GitNexus không chạy hoặc target path không được đăng ký trong knowledge graph, bước bị bỏ qua, findings vẫn giữ nguyên (không có `ast_context`).
- **Resolve**: nếu AI provider chưa cấu hình, hệ thống tự động rơi về `use_mock` hoặc sinh mock resolutions cho tất cả findings.
- **Report**: nếu không có file threshold, bước chỉ viết JSON/SARIF/Markdown bình thường.

Nhờ đó, hệ thống luôn trả về kết quả, không fail hoàn toàn.

### 3.2.4. Luồng dữ liệu chính

Một phiên quét (scan session) đi qua bốn giai đoạn tuần tự:

```
1.  Người dùng nhập đường dẫn dự án (CLI `vexcode analyze` hoặc Dashboard `Scan Modal`)
2.  Node.js (Express) kiểm tra workspace boundary, spawn Python subprocess qua `child_process.spawn`
3.  Python pipeline thực hiện pipeline 5 giai đoạn:
    Scan → Dedup → Enrich → Resolve → Report & Quality Gate
4.  Mỗi giai đoạn ghi progress JSON ra stdout (qua `emit_progress`),
    Node.js parse từng dòng và chuyển tiếp đến web dashboard qua SSE
5.  Python ghi file JSON kết quả vào `~/.vexcode/reports/{projectName}/`
    (kèm SARIF sidecar `.sarif` nếu không dùng `--no-sarif`)
6.  Node.js đọc file JSON, trả về phản hồi REST API cho dashboard
7.  Người dùng duyệt findings, xem diff, apply/reject/ignore
```

### 3.2.5. Bảng tổng hợp kiến trúc ba tầng

| Tầng | Công nghệ | Vai trò | Output |
|------|-----------|---------|--------|
| Giao diện | React 19 + TypeScript 5 + Vite 6 | Dashboard, Issues, diff view, Settings | DOM / fetch API |
| CLI / API | Node.js ESM 18.3+ + Express 4 | Điểm vào, REST API, spawn Python, progress streaming, quản lý process | REST API, SSE, child stdout |

Kiến trúc ba tầng giúp mỗi tầng phát triển và kiểm thử độc lập. Tầng Python không phụ thuộc vào Node.js; web dashboard có thể kết nối bất kỳ backend nào cung cấp đúng REST API.

### 3.2.6. Ràng buộc và hợp đồng giữa các tầng

Mỗi tầng chỉ giao tiếp với tầng liền kề thông qua interface đã định nghĩa, tuân thủ nguyên tắc "dependency inward":

| Từ tầng | Đến tầng | Giao thức | Ràng buộc |
|---------|----------|-----------|-----------|
| Giao diện | CLI / API | REST API (JSON) | Chỉ đọc kết quả, không gọi Python trực tiếp |
| CLI / API | Phân tích | stdio (JSON) qua `child_process.spawn` | Chỉ gửi `--target`, `--output`, `--mock-scan`, `--mock-ai`, `--fast`, `--no-sarif`, `--thresholds`, `--format`; nhận progress JSON từ stdout |
| Phân tích | CLI / API | File JSON + exit code | Ghi report vào `~/.vexcode/reports/`; exit code 0 = success, 1 = threshold fail |

**Quy tắc quan trọng:**
- Tầng Python không biết về Express, React hay SSE. Nó chỉ đọc env/config, chạy pipeline, ghi file, in progress ra stdout.
- Tầng Node.js không đọc nội dung file source để phân tích. Nó chỉ quản lý process, forward progress, và phục vụ file báo cáo cho frontend.
- Tầng Web không gọi Python trực tiếp. Tất cả đầu vào phải qua Express API.

### 3.2.7. State machine của Scan Session

Mỗi phiên quét có vòng đời rõ ràng, quản lý bởi `activeScan` trong bridge:

```mermaid
    [*] --> IDLE
    IDLE --> RUNNING: Start scan
    RUNNING --> COMPLETED: Python exit 0
    RUNNING --> CANCELLED: User cancel
    RUNNING --> FAILED: Python exit != 0
    COMPLETED --> [*]
    CANCELLED --> [*]
    FAILED --> [*]
```

| Trạng thái | Diễn giải | Hành vi hệ thống |
|-----------|-----------|------------------|
| `idle` | Chưa có scan nào chạy | Cho phép bắt đầu scan mới |
| `running` | Scan đang thực thi | Lưu `activeScan` record; cho phép cancel |
| `completed` | Scan thành công | Đọc report JSON, hiển thị kết quả |
| `cancelled` | Người dùng hủy | Kill process tree, xóa `activeScan`, báo cho UI |
| `failed` | Python crash | Trả về stderr, ghi log, hiển thị lỗi trên UI |

**Xử lý cancel:** Khi người dùng nhấn cancel, bridge gọi `child.kill()` trên root Python process. Node.js không kill từng subprocess: nó kill process group để đảm bảo toàn bộ cây con (ví dụ: OpenGrep binary spawn bởi Python) cũng bị terminate.

**SSE progress contract:** Mỗi dòng stdout bắt đầu bằng `{` được parse thành SSE event với format:
```json
{"type": "progress", "phase": "scan", "message": "...", "current": 1, "total": 5}
```
Dòng không phải JSON được gửi như `{"type": "stdout", "line": "..."}`. Stderr được forward riêng với `type: "stderr"`.
