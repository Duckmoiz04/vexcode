# Chương 3: Phân tích thiết kế hệ thống

## 3.1. Phân tích bài toán và yêu cầu

Chương 2 đã đặt ra bài toán Code Review tự động. Mục này xác định yêu cầu chức năng và phi chức năng cho hệ thống VexCode, đồng thời chốt hướng tiếp cận kiến trúc.

### 3.1.1. Yêu cầu chức năng

Hệ thống cần thực hiện bốn giai đoạn xử lý chính theo pipeline tuần tự:

- **Scan**: Quét mã nguồn và loại trùng lặp
- **Enrich**: Bổ sung ngữ cảnh AST
- **Resolve**: Phân loại và sinh mã sửa bằng AI
- **Report**: Tổng hợp kết quả và kiểm tra ngưỡng

Sau khi pipeline sinh báo cáo, người dùng xem và tương tác với kết quả qua giao diện web:

- **View & Act**: Duyệt findings, xem diff, apply/rollback bản sửa.

### 3.1.2. Yêu cầu phi chức năng

- **Khả năng chịu lỗi (Reliability):** Mỗi thành phần phụ thuộc đều có cơ chế fallback độc lập; hệ thống luôn trả về kết quả, không bao giờ fail hoàn toàn.
- **Hiệu năng (Performance):** Giảm chi phí và thời gian gọi AI bằng cách lọc findings qua SAST trước; lưu cache kết quả AI cho các lần quét sau; chế độ fast scan chỉ quét các file đã thay đổi.
- **Bảo mật (Security):** Kiểm tra workspace boundary trước khi đọc/ghi file; không lưu API key hardcode trong code; hỗ trợ cấu hình nhiều AI provider khác nhau.
- **Khả năng mở rộng (Extensibility):** Thêm scanner hoặc AI provider mới không cần sửa logic cốt lõi của pipeline; cấu trúc theo từng giai đoạn rõ ràng.
- **Tương thích (Compatibility):** Xuất báo cáo định dạng SARIF 2.1.0 để tích hợp với CI/CD; xuất JSON cho dashboard và CLI tiêu thụ.

### 3.1.3. Mô hình chất lượng

Hệ thống đánh giá mã nguồn dựa trên bốn đặc tính của ISO/IEC 25010:

- **Security**: Lỗ hổng bảo mật, secret lộ, CVE dependency.
- **Reliability**: Độ phức tạp cao, code dễ gây lỗi runtime.
- **Performance Efficiency**: Đo lường qua complexity metrics.
- **Maintainability**: Naming audit, duplication detection, code smell.

### 3.1.4. Lý do chọn hướng tiếp cận Hybrid SAST-first

Các công cụ AI-native (CodeRabbit, Qodo, Claude Code) dùng LLM tự quét code. Ưu điểm là linh hoạt, nhưng nhược điểm lớn là chi phí token cao, độ chính xác thấp với lỗi đã biết, và khó áp dụng chính sách nhất quán.

VexCode chọn **SAST làm lớp phát hiện chính**, AI chỉ đánh giá và đề xuất sửa. Sai số của SAST (false positive) được AI giải quyết. Phương án này giúp:
- Thu hẹp không gian tìm kiếm: từ hàng trăm nghìn dòng code xuống vài chục findings.
- Giảm chi phí: chỉ gửi đến AI những finding đã lọc qua pattern-based detection.
- Tăng độ tin cậy: luật SAST ổn định, AI phụ trợ, không phụ thuộc hoàn toàn vào LLM.

**Dễ mở rộng theo từng giai đoạn.** Pipeline chia thành bốn giai đoạn độc lập: Scan (bao gồm Dedup), Enrich, Resolve, Report: nối tiếp nhau qua interface đã định nghĩa. Mỗi giai đoạn có thể thay thế hoặc bổ sung mà không ảnh hưởng giai đoạn khác. Ví dụ:
- Thêm scanner mới (Bandit, Semgrep thuần) chỉ cần viết module mới, không cần sửa pipeline orchestrator.
- Thêm AI provider mới (NVIDIA, Cohere) chỉ cần cấu hình thêm trong `.env`, không cần đổi logic gọi AI.

### 3.1.5. Giới hạn của phiên bản hiện tại

- Chưa có cơ chế tự động kiểm tra lại logic của chương trình sau khi áp dụng bản sửa (không chạy lại test suite trên file đã sửa).
- Chưa tích hợp tự động với CI/CD (GitHub Actions, GitLab CI) để chạy quét khi có thay đổi mã.
- Chưa có trang dashboard theo dõi xu hướng lỗi theo thời gian; hiện chỉ so sánh thủ công giữa các file báo cáo JSON.
- Chưa kết hợp thêm các tiêu chí khác vào chất lượng (tỷ lệ coverage của unit test, mutation score).

### 3.1.6. Kế hoạch đánh giá (liên kết Chương 4)

Các hạn chế trên sẽ được phân tích chi tiết trong Chương 4: Thử nghiệm và đánh giá:
1. Mức độ đáp ứng từng yêu cầu chức năng và phi chức năng.
2. Nguyên nhân còn thiếu (giới hạn thời gian, phụ thuộc bên ngoài, ưu tiên độ phức tạp).
3. Đề xuất cải tiến cho các phiên bản sau của hệ thống.
