## Context and Goals

Static analysis tools like Semgrep are excellent at identifying line-level security rules, but they lack awareness of broader class hierarchies, dependencies, function signatures, and caller patterns. GitNexus builds a local repository knowledge graph that tracks these structural elements. 

In this phase, we connect the Python Analysis Core to the GitNexus CLI tool so that when a finding is reported at a specific file and line number:
1. The enclosing symbol (e.g. `Function`, `Method`, `Class`) is resolved via Cypher queries.
2. The symbol's implementation code and caller relationships are retrieved.
3. The symbol's blast radius (dependents) is calculated.
4. This gathered context is fed directly to the 9router AI completion model to construct a highly localized and safe remediation suggestion.

### In-scope
- Verification of `gitnexus` installation and index status for the target codebase.
- Development of a Python module (`ast_graph.py`) to orchestrate GitNexus CLI commands (`gitnexus cypher`, `gitnexus context`, `gitnexus impact`) and parse their outputs.
- Mapping static findings (filepath + line) to the most specific enclosing symbol node.
- Retrieving context and upstream blast radius for target symbols.
- Updating `ai_resolver.py` to compile dynamic prompts including code, callers, and blast radius.
- Standard fallback handling (revert to default scanning/resolving if GitNexus is unavailable or the repo is not indexed).

### Out-of-scope (for Phase 3)
- Automating GitNexus database initialization/rebuilding from Python Core (indexing should be run manually or via CLI/web hooks).
- Web UI visualization of the AST graph (deferred to Phase 4).
- Multi-repository cross-graph queries (deferred to future expansion).

## Architecture Decisions

### AD-01: GitNexus Subprocess CLI Calls with Markdown Parsing
- **Decision**: Execute `gitnexus` commands via Python's `subprocess.run` and parse the markdown tables inside the JSON output.
- **Rationale**: GitNexus provides a high-level CLI interface returning JSON packages where raw Cypher queries populate a `markdown` field. Parsing the markdown table in Python is extremely simple and avoids dependency on a native Node/JS binding within the Python Core.
- **Implication**: Any changes to GitNexus CLI table formatting could affect parsing. We will write a robust, error-tolerant parser that filters and strips headers.

### AD-02: AST Context Enrichment at Finding Level
- **Decision**: Keep AST context nested within the `findings` array elements rather than separate, linking them via symbol IDs.
- **Rationale**: Keeps the output schema simple and allows the Node Express API and Web UI (in Phase 4) to easily associate finding cards with their code context, callers, and impact properties.
- **Implication**: `analysis_report.json` file size will grow, but it remains fully manageable (<5MB for typical codebases).

## Component Details

### ast_graph.py (New Module)
- **Responsibilities**:
  - Run CLI check commands to verify `gitnexus` exists.
  - Determine registered repo name for the target directory (e.g. check `gitnexus list` outputs).
  - Run Cypher queries:
    `gitnexus cypher -r <repo> "MATCH (n) WHERE n.filePath = '<file>' AND n.startLine <= <line> AND n.endLine >= <line> RETURN n.id, n.name, labels(n)"`
  - Parse markdown tables returned by Cypher queries to extract the node `id`, `name`, and `label`.
  - Fetch symbol context:
    `gitnexus context -r <repo> -u <symbol_id> --content`
  - Fetch blast radius:
    `gitnexus impact -r <repo> -d upstream --depth 3 <symbol_id>`

### main.py (Modified)
- **Responsibilities**:
  - Verify GitNexus status of target directory.
  - Iterate through findings, call mapping functions in `ast_graph.py`, and populate the new `ast_context` dictionary on findings.

### ai_resolver.py (Modified)
- **Responsibilities**:
  - Check for `ast_context` in unique findings.
  - Dynamically format LLM instructions to incorporate:
    - Symbol source code.
    - Direct callers list.
    - Blast radius summary (downstream/upstream impact nodes).
